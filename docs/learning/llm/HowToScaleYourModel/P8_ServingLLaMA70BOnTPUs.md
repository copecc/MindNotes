# LLaMA 3-70B 服务

LLaMA 3-70B 的服务问题，本质上是一个容量、带宽和并行度共同约束的优化问题。它不是单纯选“更快的芯片”，而是要同时满足参数装载、`KV cache` 容量、每步延迟和跨芯片通信四个条件。

## 模型与硬件

LLaMA 3-70B 的关键超参数是：

- `L = 80`
- `D = 8192`
- `F = 28672`
- `N = 64`
- `K = 8`
- `H = 128`

对 TPU v5e，最重要的现实约束是每芯片只有 `16GB HBM`。因此，服务是否可行首先取决于两件事：

- 参数是否能放下。
- `KV cache` 是否能放下。

在这个规模下，v5e 往往比更大的芯片更有性价比，因为 inference 的主瓶颈常常是 `FLOPs / dollar`，而不是绝对峰值算力。

## KV cache 容量

若 `KV cache` 用 int8 存储，每个 token 的大小为：

$$
2 \cdot K \cdot H \cdot L = 2 \cdot 8 \cdot 128 \cdot 80 = 160\text{kB}
$$

这个数值很大，意味着上下文长度一长，batch 就会迅速受内存限制。比如在 `8192` token 上下文下，单个序列的 cache 就已经是 GB 级。

这带来两个直接结论：

- 生成阶段的 batch 上限往往由 `KV cache` 决定，而不是由参数量决定。
- 选择更小的 topology 不一定更快，因为太小会把 batch 压得过低，反而吃不满带宽和 MXU。

## 生成延迟

对于 decode，常用的近似下界是把每步看作“从 HBM 读参数和 `KV cache`，再做计算”。当 batch 不大时，这个估计通常已经很接近真实值。

在一组典型配置里，batch `32`、上下文 `8192`、参数和 KV 都用 int8 时：

- 总参数约 `70GB`。
- `KV cache` 约 `41.9GB`。
- 总内存约 `112GB`。

因此最小可用 slice 大约是 `4x2`，而更稳妥的选择是 `4x4`。在这个配置下，decode 步长的理论下界大约是 `17ms`，对应约 `235 tokens/s/chip`。如果拓扑扩大到 `4x4`，延迟大约减半，但单 chip 吞吐不一定按同样比例变化，因为瓶颈仍主要在 HBM 与 `KV cache`。

## 吞吐边界

对 bf16 matmul，在 `B \ll D, F` 时，critical batch 约为 `240 tokens`。如果只把参数量化到 int8，而计算仍用 bf16，则临界 batch 约降到 `120`。如果权重和计算都走 int8，则硬件峰值也变了，临界 batch 又会回到大约 `240`。

这说明量化的收益不只是“参数更小”，而是同时改变了：

- 每次搬运的字节数。
- 可用计算峰值。
- 能支持的 batch 上限。

对 LLaMA 3-70B，真正限制生成吞吐的往往不是单步 FLOPs，而是 `KV cache` 和 HBM 带宽。若 batch 上不去，MXU 峰值就很难被吃满。

## topology 选择

只看“能不能放下”时，最小 slice 可能是：

- bf16：`4x4`
- int8：`4x2`
- int4：`2x2`

但最小 slice 不一定是最优 slice。随着 topology 变大，`KV cache` 可容纳的 batch 会增加，生成吞吐反而可能上升。也就是说，服务推理时要找的是性能最优点，而不是显存刚好放下的最小点。

## 并行与分工

`Prefill` 和 `Generation` 的最优并行方式不同。

`Prefill` 更接近训练，通常可以先做模型并行，再在必要时做序列并行。

`Generation` 更受通信和缓存约束，通常只能依赖模型并行，而且并行度不能无限增加。对 LLaMA 3-70B，在 bf16 配置下，纯模型并行一般很难直接把 topology 推到 `4x8` 仍保持高吞吐；量化之后，空间会宽松一些。

这也是为什么推理系统经常把前后两阶段拆开：前向 prefilling 更适合高计算密度，decoding 更适合低通信和高缓存效率。

## 服务策略

实际服务中，最重要的三个结论是：

- `Prefill` 和 `Generation` 应当分开优化。
- `KV cache` 的规模决定了 batch 和 topology 的上限。
- 通过减少 `KV cache`、提高 topology 或引入更强量化，通常比单纯追求更高峰值算力更有效。

在一个稳定的服务系统里，`Prefill` 服务器和 `Generation` 服务器还需要按负载比例配置。对这类典型设定，`Prefill` 侧通常要比 `Generation` 侧更多，因为前者是长计算、后者是长尾串行步骤。

## 工程结论

LLaMA 3-70B 的服务不是“把模型塞进 TPU”这么简单，而是一个多目标平衡问题：

- 容量上，参数和 `KV cache` 共同决定能否部署。
- 延迟上，decode 主要受 HBM 和 `KV cache` 影响。
- 吞吐上，batch 能否被撑大比单卡峰值更关键。
- 并行上，`Prefill` 与 `Generation` 应当分工，而不是强行共享同一套配置。

如果只记一个结论，那就是：在 70B 级别的生成服务里，`KV cache` 往往比参数更先决定系统形状。

## 补充算例

- 在 `bfloat16` 的 `4x8` 配置下，参数之外仍可留下约 `372GB` 给 KV cache，batch 可提高到约 `140`，对应 `0.44 QPS/chip`。
- 在 `int8` 的 `4x4` 配置下，约可达到 `0.90 QPS/chip`；在 `int4` 的 `2x4` 配置下，约可达到 `1.80 QPS/chip`。
- 这些数值直接说明：最小可行拓扑不一定是吞吐最优拓扑，KV cache 容量才是决定 batch 和吞吐的关键约束。
