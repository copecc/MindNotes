# 结论与延伸

这里不再继续推导新的性能公式，而是把 roofline、TPU 执行路径、分片矩阵和 JAX 并行模型压缩成一条后续阅读路线。它们可以作为继续分析系统问题的起点。

## 收束

下列分析链条适合作为继续展开问题时的固定框架：

- 先用 roofline 判断瓶颈是计算、片内带宽还是跨设备通信。
- 再看硬件执行单元和数据路径，确认算子到底落在 MXU、VPU 还是网络层。
- 然后把分片方式映射到 collectives，判断 `AllGather`、`AllReduce`、`ReduceScatter` 或 `AllToAll` 的代价。
- 最后再回到 JAX 或其他编程框架，决定是让编译器自动处理，还是显式写出通信。

这种分析方法的价值主要在于把这四层连起来，而不是依赖某一个孤立公式。

## 阅读路线

如果继续深入，后续阅读大致可以分成几类：

- TPU 架构与 kernel 编程
- JAX 的分布式数组与自动并行
- GPU / CUDA 的矩阵乘法优化
- LLM 训练中的并行策略和网络拓扑

这些材料的侧重点不同，但问题都相同：如何让计算、存储和通信的瓶颈尽量落在你能控制的那一层。

## 结论

核心信息可以压缩为：

- 理论公式只能指出方向，不能替代对执行路径的理解。
- 具体硬件上的峰值吞吐，最终要通过数据布局和通信模式兑现。
- TPU、GPU、JAX、Pallas、NCCL 这些工具栈的差异，主要体现在“谁负责决定通信”。

因此，这份笔记更接近一份索引。真正有用的部分不在致谢信息，而在于后续阅读时可以沿哪些层次继续拆解问题。
