# JAX 上的 TPU 编程

JAX 的多设备编程可以分成三层：交给编译器自动切分，显式声明张量如何分片，或者把通信原语也写成程序的一部分。三者的区别不在语法，而在于谁来决定数据布局和通信位置。

## 并行模式

`jax.jit` 配合 sharded 输入时，XLA 可以自动补齐 `AllGather`、`ReduceScatter`、`AllReduce` 等通信。这一模式适合先把单卡程序搬到多卡，但代价是通信由编译器推断，错误布局也更难显式控制。

显式分片把 sharding 变成类型信息。张量的局部形状、mesh 轴和输出布局都要被明确写出，因此 JAX 可以在分片语义不清时直接报错。这一模式比自动分片更可控，但仍然不要求程序员手写 collectives。

`jax.shard_map` 则直接给出设备局部视角。程序在每张卡上按局部 shard 执行，所有跨卡数据交换都要显式写出。它的优点是通信语义最清楚，缺点是代码更接近底层实现。

可以把三种模式概括为：

- 自动分片：让编译器决定怎样并行。
- 显式分片：让 JAX 解释布局，但仍自动补通信。
- 手工分片：程序员直接指定通信。

## 自动分片

在自动模式下，`jax.jit` 仍然像普通 JIT 一样编译函数，但如果输入或输出带有 shard 信息，XLA 会在 HLO 层自动插入 collective。一个典型例子是把输入激活分布在 `P('X', 'Y')`，把权重放在 `P('Y', None)`，随后让输出按 `P('X', None)` 排布。对 `In[B_X, D_Y] @ W[D_Y, F] -> Out[B_X, F]` 这种布局，编译器通常会在 matmul 后加入 `AllReduce`。

这类自动化能把“先写单卡代码，再扩到多卡”的成本压得很低，但也意味着：

- 只要分片语义稍有歧义，编译器可能选择次优通信。
- 局部计算和通信的重叠是否充分，往往需要靠 profiling 复核。
- 对需要精确控制内存峰值的程序，自动插入的中间量可能并不理想。

## 显式分片

显式分片的重点不是执行路径，而是把“哪个维度在哪条 mesh 轴上切分”写清楚。这样做的直接收益是，分片是否合法、是否需要规约、是否需要收集，都可以在类型层面提前暴露出来。

对矩阵乘法来说，最关键的是求和维是否完整：

- 求和维完整时，本地 matmul 就是完整子结果。
- 求和维被切开时，本地结果只是 partial result，必须再规约。

JAX 在这种模式下的价值，是把这些分片约束提前变成显式错误，而不是等到运行时才发现结果布局不对。

## 手工通信

`jax.shard_map` 适合需要把通信路径也纳入性能设计的场景。它允许你直接把局部循环、块级 matmul 和 collectives 组合起来，例如：

- 需要完整张量时，显式 `all_gather`
- 需要跨设备求和时，显式 `psum`
- 需要重排 token 或 expert 时，显式 `all_to_all`

这一模式的核心价值是可预测性。对于 TPU 上常见的大矩阵和 MoE 负载，是否把通信放在前向之前、块间、还是后向中间，往往直接决定是否能把 HBM 或 ICI 的等待隐藏掉。

## 练习的作用

这类练习并不是为了展示 API 技巧，而是在训练一个判断顺序：

1. 先确定本地 shard 能否独立完成算子。
2. 再确定缺失的是输入、输出，还是路由信息。
3. 然后选择 `all_gather`、`psum`、`all_to_all` 之类的原语。
4. 最后再问通信能否和局部计算重叠。

这个顺序与 roofline 分析一致。JAX 代码写法只是外壳，真正决定性能的是数据如何跨 shard 流动，以及这段流动是否被 MXU 或 ICI 的计算窗口掩盖。

## 工程结论

可以把结论压缩为：

- 自动分片适合快速扩展，但需要 profiling 验证通信位置。
- 显式分片适合表达清楚布局意图，并提前暴露歧义。
- `shard_map` 适合把通信当成第一等公民来优化。
- 对 TPU 而言，真正的性能收益通常来自减少 blocking `AllGather`，以及把 collectives 和块级 matmul 流水化。

## 补充算例

- 在 `X * Y = N` 的分片布局下，如果目标只是对每个 shard 内的激活求平均，计算可以完全局部完成，不需要额外的 collective。若 `roll` 只发生在 shard 内部，同样不需要跨设备交换。

- MoE 路由时，真正昂贵的步骤通常不是专家矩阵乘法本身，而是把激活先聚齐再分发。若路由只命中少量专家，使用显式 `AllToAll` 处理稀疏 token，通常比先构造大而密的中间张量更省显存，也更省通信。

- 对 collective matmul，如果 contracting 维被切开，朴素做法往往会先做本地乘法再规约。更好的做法是把输出维切成块，让本地计算和 `psum`/`AllReduce` 交叠；如果通信改成双向，尾部等待还可以进一步缩短。
