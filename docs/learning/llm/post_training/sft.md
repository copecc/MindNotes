# SFT

!!! abstract

    监督微调（Supervised Fine-Tuning, SFT）是大语言模型后训练阶段最常见的第一步。它通过带标注的输入-输出样本，把预训练模型从“擅长续写通用文本”调整为“更稳定地完成特定交互任务”。其核心不在于向模型注入全新的语言能力，而在于重新分配条件概率，使目标风格、任务格式、指令遵循方式与工具调用模式在推理时更容易被激活。

## 目标

SFT 的直接目标，是在给定输入条件 $x$ 时，提高目标响应 $y$ 的条件概率 $P(y \mid x)$。这里的“目标响应”可以是问答结果、结构化输出、代码补全、工具调用序列，也可以是多轮对话中某一轮 assistant 应答。

从能力结构看，预训练已经提供了语言建模、世界知识压缩、常见推理模式与文本风格生成能力；SFT 主要完成以下调整：

- 强化指令跟随，使模型优先响应显式任务要求。
- 强化角色边界，使 system、user、assistant 等消息在推理时具有更稳定的语义分工。
- 强化输出格式，使 JSON、XML、函数调用参数、分类标签等目标形式更易生成。
- 强化领域分布，使模型在特定语料风格、术语和任务模式上具有更高命中率。
- 强化行为先验，使拒答、澄清、工具调用、引用格式等策略更一致。

因此，SFT 更接近“行为分布重塑”而不是“能力凭空生成”。如果目标任务所需知识、推理链条或外部工具使用方式在基座模型中完全缺失，仅靠有限规模 SFT 通常难以稳定补齐。

## 监督目标

SFT 通常仍沿用自回归语言模型的最大似然训练框架。给定样本对 $(x, y)$，训练目标是最大化目标序列在条件上下文下的联合概率：

$$
\max_\theta \log P_\theta(y \mid x) = \max_\theta \sum_{t=1}^{T} \log P_\theta(y_t \mid x, y_{<t})
$$

其中 $\theta$ 为模型参数，$y_t$ 为目标序列第 $t$ 个 token。对应的最小化形式通常写为交叉熵损失：

$$
\mathcal{L}_{\mathrm{SFT}} = - \sum_{t \in \mathcal{M}} \log P_\theta(z_t \mid z_{<t})
$$

这里 $z$ 表示拼接后的完整训练序列，$\mathcal{M}$ 表示参与损失计算的位置集合。$\mathcal{M}$ 的定义决定了哪些 token 被视为监督目标，这正是 assistant-only loss masking、packing 与多轮对话处理的关键。

从优化角度看，SFT 与预训练共享同一类 token-level 目标，但样本分布、损失掩码、模板结构和目标行为不同。因此，二者在形式上相似，在数据语义上差异很大。

## Token 级目标

### 下一 token 预测仍是基本单元

即使样本表现为“一条指令对应一段回答”，模型内部优化的最小单位仍然是单个 token。训练时并不存在一个整体级别的“回答正确”标量监督，而是将目标回答拆成 token 序列，逐位置拟合其条件概率。

这意味着 SFT 的效果同时受以下因素影响：

- 目标答案是否唯一。如果可接受答案很多，而数据只保留其中一种表达，模型会被推向更窄的表述分布。
- 序列前缀是否稳定。模板、角色标记和工具格式的微小差异会改变后续 token 的条件分布。
- 长回答中的后部 token 是否足够可靠。若尾部噪声较高，模型会学习不稳定续写模式。
- 高损失位置是否集中。结构化字段、稀有术语、长数字串和代码符号通常更难拟合。

### 序列级行为如何由 token 级目标形成

SFT 常希望得到“会遵循指令”“会调用工具”“会先思考再回答”这类宏观行为，但优化器只能看到 token 级误差。序列级行为之所以出现，是因为某些 token 片段在特定上下文中被反复强化，最终形成稳定生成路径。

例如，在工具使用数据中，若模型多次见到：用户提问、assistant 输出工具名与参数、tool 返回结果、assistant 给出整合回答，那么它学习到的并不是抽象规则本身，而是这类状态转移在 token 空间中的高概率轨迹。

因此，SFT 数据设计的重点通常不只是“答案对不对”，还包括“上下文触发条件是否清晰”“中间结构是否规范”“目标行为是否可由模板稳定对齐”。

## Assistant-Only Loss Masking

### 基本动机

在对话数据中，一条训练样本常包含 system、user、assistant，甚至 tool、observation 等多种角色。如果对整条序列所有 token 都计算损失，模型会同时学习“复述用户输入”和“预测系统提示”，这通常不符合训练目标。

assistant-only loss masking 的做法，是只对 assistant 目标片段计算损失，其他角色 token 仅作为条件上下文输入模型，不参与梯度更新。

这样做有三个直接作用：

- 保留完整上下文，使模型仍能利用 user 与 system 信息进行条件生成。
- 避免将训练信号浪费在复制提示词或用户文本上。
- 使优化更聚焦于“在给定上下文下 assistant 应该说什么”。

### 典型掩码方式

常见做法是先按 chat template 将多轮消息线性化，再根据角色边界生成 label mask：

- system token：mask 掉，不计损失。
- user token：mask 掉，不计损失。
- assistant token：保留标签，计入损失。
- 特殊分隔符：按实现选择保留或 mask，关键是训练与推理模板保持一致。
- tool 或 observation token：是否计损失取决于训练目标。

若目标是训练模型“生成工具调用”，通常会对 assistant 中的 tool-call 结构计损失；若目标是训练模型“消费工具结果再回答”，工具返回内容本身常作为上下文而非监督目标。

### 边界与副作用

assistant-only masking 并不意味着 user token 无关。它们仍决定隐藏状态分布，因此模板设计、角色标记和上下文顺序仍然会显著影响训练效果。

另一方面，若 assistant 回复中混入大段引用用户原文、日志文本或工具原始输出，模型依然会在这些 token 上承受监督压力，结果可能是复制倾向增强。因此，是否保留原样引用、是否压缩 observation、是否把中间结构放入 assistant 角色，都会改变训练结果。

## 单轮、多轮与对话建模

### 单轮样本

单轮样本通常形如“指令 + 单次回答”。这种形式数据清洗和模板构造较简单，适合：

- 通用 instruction tuning
- 分类、抽取、改写等局部任务
- 结构化输出学习
- 较短上下文的领域问答

其局限在于，对话状态转移较弱。模型能学会“收到某类请求时如何答复”，但不一定能稳定处理追问、上下文修正、角色记忆和跨轮工具协同。

### 多轮样本

多轮样本把历史消息作为同一条训练序列输入，使模型学习上下文累积后的响应方式。它更适合：

- 澄清与追问
- 多步任务拆解
- 带工具的交互流程
- 受 system policy 约束的长对话

多轮训练的难点不在形式，而在数据质量。若历史轮次存在冗余、冲突或角色格式不稳定，模型会学习到噪声状态转移。常见问题包括：

- 某些轮次 assistant 回复质量明显低于最终轮，导致前缀状态劣化。
- 同一任务在不同样本中采用不同模板，破坏角色边界的一致性。
- 长上下文中有效监督比例下降，计算成本显著增加。

### 单轮与多轮的关系

二者不是替代关系，更常见的是混合使用。单轮样本提供高密度监督，多轮样本提供状态转移模式。训练配比需要根据目标产品决定：若模型主要用于一次性问答或离线改写，单轮占比通常更高；若模型主要用于 agent、客服或 copilot，对多轮和工具流程的依赖会更强。

## Chat Template

### 为什么模板重要

聊天模型训练并不是直接把结构化消息送入 Transformer，而是先通过 chat template 把消息序列展开为线性 token 流。模板决定了：

- 角色如何标记
- 轮次如何分隔
- assistant 起始位置如何提示
- 是否插入 BOS、EOS、special token
- 工具调用、工具返回和结构化字段如何编码

因此，template 是“数据语义”到“token 序列”的映射层。训练与推理模板不一致，常导致模型行为显著退化，例如角色混淆、停止位置异常、工具格式偏移、回答风格漂移。

### 常见模板差异

不同模型族常采用不同的角色标记方案，例如：

- 纯文本分隔符风格
- 显式 special token 风格
- 带 header 的 instruction 风格
- 原生支持 tool、function、json schema 的扩展风格

这些差异会影响模型对边界的感知强度。显式 special token 通常更利于稳定解析角色切换；纯文本模板更灵活，但更依赖数据一致性与 tokenizer 兼容性。

### 模板设计原则

模板设计通常遵循以下原则：

- 角色边界明确，避免 user 与 assistant 片段混淆。
- 训练与推理完全一致，避免线上分布偏移。
- 特殊 token 尽量少而稳定，避免引入不必要的稀疏模式。
- 工具与结构化字段有明确起止标记，便于模型学习格式约束。
- 停止条件可控，避免 assistant 回答越界到下一角色。

## Packing

### 基本目的

packing 指把多条较短样本拼接到同一个定长训练序列中，以减少 padding 浪费并提升吞吐。对于大量短指令数据，若不做 packing，GPU 计算会被无效填充显著稀释。

### 训练收益

packing 的主要收益包括：

- 提高 token 利用率，减少空白 padding。
- 提高每步有效监督密度。
- 在固定显存下容纳更多真实 token。
- 对短样本占比高的数据集更有效。

### 需要控制的问题

packing 会引入样本边界管理问题。如果简单拼接而不做边界隔离，后一个样本的前缀可能错误地看到前一个样本的尾部，形成不真实的跨样本依赖。常见处理方式包括：

- 在样本之间插入 EOS 或专用分隔符。
- 在 attention mask 中阻断跨样本可见性。
- 在 loss mask 中仅保留每个样本各自的有效目标区间。

对纯 causal LM 而言，若只插入 EOS 而不阻断注意力，模型仍能看到前一条样本。某些实现接受这种近似，以换取更简洁的数据流水线；某些实现则采用 block-diagonal attention 维持严格样本独立。二者是工程折中，而不是统一标准。

## SFT 的几种常见类型

### 对比

| 类型 | 核心目标 | 主要数据形态 | 典型收益 | 主要风险 |
| :--- | :--- | :--- | :--- | :--- |
| Instruction Tuning | 强化指令跟随与通用助手风格 | 指令-回答、单轮问答、多轮聊天 | 通用交互质量提升，格式更稳定 | 容易过度收缩表达分布，通用知识收益有限 |
| Domain Tuning | 强化特定领域术语、任务与风格 | 领域问答、专业文档改写、结构化任务 | 领域命中率与术语一致性提升 | 领域过拟合，通用能力和安全边界可能漂移 |
| Tool-use SFT | 强化工具选择、参数构造与结果整合 | 调用轨迹、函数参数、tool observation、多轮 agent 流程 | 工具调用更稳定，外部能力接入更顺畅 | 模板脆弱，错误轨迹会固化为行为先验 |

### Instruction Tuning

instruction tuning 关注通用任务指令遵循，是大多数聊天模型的基础 SFT 形式。其数据往往覆盖开放问答、总结、翻译、改写、分类、推理表达和安全拒答等多种任务。

它的价值主要体现在交互层而不是知识层。模型原本可能“知道”某件事，但不一定会按用户要求给出简洁、结构化、合规的回答；instruction tuning 负责把这些行为模式组织成稳定输出。

### Domain Tuning

domain tuning 的目标是把模型分布向特定领域推进，例如法律、医疗、金融、代码、企业内部知识库或某一类写作风格。

其关键不只是加入更多领域文本，而是让输入条件与目标行为共同体现领域任务结构。例如，法律领域可能需要条文引用与争点拆分，代码领域可能需要 API 约束与补全格式，企业知识场景可能需要固定术语与内部模板。

如果领域数据规模大且风格单一，domain tuning 容易提升专业表达稳定性；但若数据过窄，也更容易造成遗忘、口吻单一或对开放任务的泛化下降。

### Tool-use SFT

tool-use SFT 训练模型在合适时机输出工具调用，并在得到 observation 后继续整合结果。它的本质，是把“调用外部系统”转化为可学习的文本或特殊 token 序列。

这类训练通常涉及三类对象：

- 触发条件：什么情况下应调用工具。
- 调用结构：调用哪个工具、参数如何组织。
- 结果消费：拿到 observation 后如何继续回答。

tool-use SFT 对模板一致性要求很高。工具名、参数序列化方式、返回内容边界、异常分支格式，只要训练与推理存在偏移，模型就可能在关键位置失稳。

## 数据构造与样本选择

### 质量比规模更敏感

SFT 的参数更新量通常远小于预训练，因此每条样本的梯度影响更直接。相比海量弱标注，较小规模但高一致性的样本往往更有效。样本质量主要体现在：

- 输入是否清晰表达任务条件。
- 输出是否满足目标风格与格式。
- 多轮历史是否必要且无冲突。
- 工具轨迹是否符合真实线上协议。
- 拒答、澄清、边界处理是否一致。

### 去重与分布控制

重复样本会放大某些模式的梯度权重，使模型对局部表达过拟合。领域样本与通用样本的混合，也需要控制配比，否则容易出现行为漂移。常见做法包括：

- 文本级与语义级去重。
- 按任务类别控制采样权重。
- 对长短样本进行长度分桶。
- 对稀有但关键模式单独增权。

### 拒答与边界样本

如果产品要求模型在某些条件下拒答、要求澄清或转交工具，相关样本必须显式进入 SFT 数据。否则模型只会学习“如何回答”，而不会稳定学习“何时不直接回答”。这类边界行为若完全依赖后续 RL 或规则层，基础分布往往不够稳。

## 常见问题

### SFT 为什么会让模型更像助手？

因为数据把大量“用户请求 -> assistant 响应”的条件映射反复呈现给模型，使 assistant 风格 token 在相应上下文中的条件概率持续上升。角色边界、语气、格式和结构并不是独立模块，而是在 token 分布中共同被强化。

### SFT 会不会教会模型新知识？

会，但程度有限。若新知识以高质量样本反复出现，模型可能记住局部事实与表达模式；但从机制上看，SFT 更擅长重排已有能力与行为优先级。对于覆盖面广、更新频繁或需要精确检索的知识，仅靠 SFT 的成本和稳定性通常不理想。

### 为什么只在 assistant token 上算损失？

因为训练目标通常是“给定上下文后 assistant 如何响应”。system 和 user token 更适合作为条件输入，而不是监督对象。若把它们也纳入损失，模型会额外学习复述提示词或用户内容，这通常会降低监督效率。

### 多轮数据一定优于单轮数据吗？

不一定。多轮能提供状态转移模式，但也更贵、更长、更容易带入噪声。若目标任务主要是一次性结构化输出，单轮高质量数据常更有效。是否采用多轮，应由目标交互形态决定，而不是由“更接近聊天”这一表面特征决定。

### Packing 会不会改变训练分布？

会。只要多条样本被放进同一序列，边界处理方式就会影响可见上下文。若不做严格隔离，模型会接触到训练时并不存在于真实任务中的跨样本邻接关系。是否接受这种偏差，取决于吞吐收益与分布保真的权衡。

### Tool-use SFT 的主要难点是什么？

主要难点不在“调用工具”这个动作本身，而在协议一致性。工具名、参数 schema、返回格式、异常路径和多轮状态必须稳定；否则模型学到的是互相冲突的轨迹，推理时很容易在格式或时机上失稳。

## 边界

SFT 在后训练体系中位置重要，但边界也很明确。

### SFT 不直接优化偏好排序

若两个回答都正确，但一个更简洁、一个更稳妥、一个更符合品牌口吻，单纯的 token-level 监督未必足以表达这种相对偏好。偏好学习、RLHF、DPO 等方法更适合处理“多个可接受答案之间如何排序”的问题。

### SFT 不保证强推理提升

对复杂推理任务，SFT 可以通过数据模式强化某些推理模板，但它并不天然提供更强的搜索、验证或规划机制。若训练数据只给出最终答案而没有稳定中间结构，模型获得的往往是结果分布调整，而不是可泛化的推理过程改进。

### SFT 不替代外部知识系统

对于时效性强、覆盖面大或必须精确引用的知识，检索增强、数据库查询和工具调用通常比把知识压进参数更稳定。SFT 更适合训练“如何使用这些外部能力”，而不是替代它们。

### SFT 容易受到数据协议影响

模板、角色标记、特殊 token、EOS 位置、工具 schema 和 observation 格式，都会直接进入训练分布。协议不稳定时，问题通常不是“模型没学会”，而是“模型学到的是混杂协议”。因此，SFT 的很多失败并不是优化器问题，而是数据接口问题。

## 要点

- SFT 的核心是重新塑造条件分布，使模型在给定上下文下更容易生成目标行为。
- 优化目标仍是自回归 token-level 交叉熵，序列级行为来自大量局部 token 轨迹的重复强化。
- assistant-only loss masking 用于把 user、system 等角色作为条件上下文，仅对 assistant 目标片段计算损失。
- 单轮数据提供高密度监督，多轮数据提供状态转移模式，二者需要根据产品交互形态组合。
- chat template 是结构化消息到 token 序列的映射层，训练与推理必须保持一致。
- packing 提高吞吐，但会引入样本边界管理问题，需要通过分隔符、attention mask 或 loss mask 控制分布偏差。
- instruction tuning、domain tuning 和 tool-use SFT 分别面向通用交互、领域行为和外部工具协议，数据形态与失败模式不同。
- SFT 适合行为对齐与格式学习，不直接等价于偏好优化、强推理增强或动态知识更新。
